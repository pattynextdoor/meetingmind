# CONTENT.md вЂ” authoring guide for meetingmind.me content

This document covers how to author and publish content on **meetingmind.me**.
Read it once end-to-end before writing your first post. After that, the
**Pre-publish checklist** at the bottom is the only thing you need to revisit.

---

## Quick orientation

The site is **two Vite projects merged at build time**:

| Path | Owner | What it serves |
|---|---|---|
| `/` | `frontend/` (Vite + React SPA) | Marketing homepage |
| `/blog`, `/blog/*`, `/about`, `/feed.xml`, `/llms.txt`, `/llms-full.txt`, `/sitemap-index.xml` | `content/` (Astro SSG) | Content routes, all server-rendered to static HTML |
| `docs.meetingmind.me` | `docs/` (VitePress, separate Vercel project) | Product documentation |

The merged build runs via `npm run build` from `frontend/` (which calls
`scripts/build-with-content.mjs`). Vercel deploys `frontend/dist`.

## Authoring a new post

### 1. Create the file

```
content/src/content/blog/<your-slug>.mdx
```

The slug becomes the URL: `/blog/<your-slug>`.

Copy `content/src/content/blog/example-post.mdx` вЂ” it exercises every feature
the system supports.

### 2. Fill in frontmatter

```yaml
---
title: "How to import Zoom transcripts into Obsidian"   # в‰¤ 60 chars
description: "Step-by-step: export Zoom .vtt, drop into MeetingMind's watched folder, get auto-linked notes."  # в‰¤ 160 chars
publishedAt: 2026-05-15
updatedAt: 2026-05-15       # optional; bump when you make material edits
author: "Patrick Tumbucon" # optional; defaults to this name
tags: [zoom, transcripts, obsidian]
draft: false                # set true to exclude from production builds
canonical: ""               # optional; only set if syndicating from elsewhere
noindex: false              # optional; true в†’ adds robots noindex
ogImage: ""                 # optional override; auto-generated by default
faq:                        # optional; emits FAQPage schema if present
  - q: "Does this work with cloud recordings?"
    a: "Yes. Zoom's cloud recordings produce a .vtt transcript that imports identically."
howto:                      # optional; emits HowTo schema if present
  name: "Import a Zoom transcript"
  description: "One-time setup, then drag-and-drop."
  totalTime: PT2M           # ISO 8601 duration
  steps:
    - name: "Download the .vtt"
      text: "From Zoom recording page, click Download в†’ Transcript."
    - name: "Drop into your watched folder"
      text: "Move the file into the folder MeetingMind watches."
---
```

**Frontmatter rules** (enforced by `content/src/content.config.ts`):

| Field | Required | Max length / format | Notes |
|---|---|---|---|
| `title` | вњ“ | 70 chars (Zod cap; check-seo enforces 60) | Goes in `<title>` and the H1 |
| `description` | вњ“ | 170 chars (Zod cap; check-seo enforces 160) | Goes in meta description, OG, Twitter, RSS |
| `publishedAt` | вњ“ | ISO date | Used in BlogPosting + sitemap |
| `updatedAt` | вЂ“ | ISO date | Bump on material edits; shown on page |
| `tags` | вЂ“ | array of strings | Used for related-posts and OG keywords |
| `draft` | вЂ“ | boolean | Drafts skipped in production builds |
| `faq` | вЂ“ | array of `{q, a}` | Emits FAQPage JSON-LD; renders as `<FAQ />` if you include `<FAQ items={frontmatter.faq} />` in the body |
| `howto` | вЂ“ | object | Same as faq вЂ” schema is auto, body component is opt-in |
| `noindex` | вЂ“ | boolean | True в†’ robots `noindex,nofollow` |
| `canonical` | вЂ“ | URL | Override only if cross-posting |
| `ogImage` | вЂ“ | image path | Override the auto-generated OG image |

### 3. Write the body

The body is **MDX** вЂ” Markdown plus Astro components.

#### Available components

```mdx
import Callout from '~/components/Callout.astro';
import FAQ from '~/components/FAQ.astro';
import HowTo from '~/components/HowTo.astro';
import YouTube from '~/components/YouTube.astro';

<Callout type="info" title="Tip">Use these for content the reader must not miss.</Callout>
<Callout type="warn">For destructive or non-obvious advice.</Callout>
<Callout type="note">For meta or background context.</Callout>

<FAQ items={frontmatter.faq} />          {/* renders inline + emits FAQPage schema */}
<HowTo {...frontmatter.howto} />         {/* renders inline + emits HowTo schema */}

<YouTube id="KZ8D_RlLz7c" title="MeetingMind demo" />   {/* responsive, lazy-loaded, no-cookie */}
```

#### Conventions

- **One H1 per page** вЂ” comes from frontmatter `title`. Don't add `# Heading` in the body.
- **Use `##` for sections** вЂ” these auto-populate the table of contents.
- **Code blocks** вЂ” fence with the language for Shiki highlighting:

  <pre><code class="language-mdx">```typescript
function example() { return 42; }
```</code></pre>

- **Images** вЂ” always include `width`, `height`, and `loading="lazy"`:

  ```mdx
  <figure>
    <img src="/path/to/image.png" alt="Descriptive alt" width="800" height="450" loading="lazy" />
    <figcaption>Optional caption.</figcaption>
  </figure>
  ```

- **Internal links** вЂ” use relative or root-relative paths (`/about`, `/blog/other-post`). The SEO check breaks the build on dead internal links.

### 4. Run it locally

```bash
cd content
npm run dev      # starts Astro on http://localhost:4321
```

Visit `http://localhost:4321/blog/<your-slug>` to preview.

## How the SEO/schema system works

### Automatic per post

- `<title>` (from frontmatter), `<meta name="description">`, `<link rel="canonical">`
- Open Graph + Twitter Card tags
- A 1200Г—630 OG image at `/blog/<slug>/og.png` (Satori-rendered from frontmatter)
- A plain-markdown variant at `/blog/<slug>.md` (referenced in `<link rel="alternate">`, exposed in `llms.txt`)
- `BlogPosting` JSON-LD (headline, description, author, dates, image, mainEntityOfPage)
- `BreadcrumbList` JSON-LD (Home в†’ Blog в†’ Post)
- Sitemap entry with `lastmod`
- RSS entry

### Automatic sitewide

- `Organization` JSON-LD on every page (`content/src/components/seo/OrganizationSchema.astro`)
- `WebSite` JSON-LD on every page
- Robots meta + canonical
- Inclusion of `/sitemap-index.xml` in `frontend/public/robots.txt` (allows all standard + LLM crawlers)

### What needs per-post attention

- **`faq` frontmatter** if your post has an FAQ section вЂ” emits `FAQPage` JSON-LD
- **`howto` frontmatter** if your post is a how-to вЂ” emits `HowTo` JSON-LD
- **`ogImage`** if you want a custom OG image (otherwise auto-generated)

### What lives in `frontend/index.html`

The marketing homepage has its own JSON-LD blocks for `Organization`,
`WebSite`, `SoftwareApplication`, `Product`, `Person`, and `FAQPage`. These are
inline `<script type="application/ld+json">` tags in
`frontend/index.html`. **The FAQPage block MUST mirror the visible FAQ in
`frontend/App.tsx`** вЂ” Google penalizes schema-only content. When you add a
question to one, add it to the other.

## Updating llms.txt

The route is generated at `content/src/pages/llms.txt.ts`. Edit the file directly when you want to:

- Add or remove curated links in the **Product**, **Documentation**, or **Contact** sections
- Change the site description shown to LLM crawlers

The **Blog** section is auto-populated from published posts вЂ” you do not need to touch llms.txt when you publish a new post. The same goes for `llms-full.txt` at `content/src/pages/llms-full.txt.ts`, which dumps full post bodies.

After editing llms.txt source, run `npm run check:llms` to validate the spec and link health.

## Verification scripts

Run from the repo root:

| Command | What it does |
|---|---|
| `npm run check:seo` | Crawl built site; flag missing meta, oversized titles/descriptions, missing canonical, broken internal links |
| `npm run check:schema` | Validate every JSON-LD block вЂ” parses, has `@context`, `@type`, type-specific required fields |
| `npm run check:llms` | Validate llms.txt against the spec; verify local links resolve |
| `npm run check:llms:network` | Also HEAD external URLs (slower; CI runs this) |
| `npm run check:all` | Runs check:seo + check:schema + check:llms in sequence |
| `npm run content:build` | Run the full merged build manually |

Each script exits non-zero on failure, so CI catches issues before merge.

## Pre-publish checklist

Before opening a PR for a new post:

- [ ] Frontmatter complete: `title`, `description`, `publishedAt`, `tags`
- [ ] Title is в‰¤ 60 chars; description is в‰¤ 160 chars
- [ ] `draft: false` (or unset)
- [ ] At least two `##` sections (so a TOC renders)
- [ ] Any FAQ section has matching `faq:` frontmatter
- [ ] Any HowTo section has matching `howto:` frontmatter with в‰Ґ 2 steps
- [ ] All images have `width`, `height`, `loading="lazy"`, and an `alt`
- [ ] Internal links resolve (`/about`, `/blog/...`)
- [ ] `cd frontend && npm run build` succeeds locally
- [ ] `npm run check:all` passes
- [ ] Spot-check the OG image at `frontend/dist/blog/<slug>/og.png`
- [ ] Open `frontend/dist/blog/<slug>.html` in a browser and skim
- [ ] Check `frontend/dist/blog/<slug>.md` is readable (LLM crawlers see this)

CI runs the full build + all checks on every PR via `.github/workflows/content.yml`.

## FAQ for authors

**Why MDX and not Markdown?** Because we want components (FAQ, HowTo, Callout) inside posts. Plain `.md` still works for prose-only posts; rename to `.md` and skip the imports.

**Why don't drafts show up in production?** The `getPublishedPosts()` helper filters `draft: true` when `import.meta.env.PROD`. In dev they appear so you can preview.

**Where do I put images?** For post-specific images, put them in `content/src/assets/blog/<slug>/` and import them in MDX so Astro can optimize. For shared assets that already live in `frontend/public/`, reference them by absolute path (`/meetingmind_logo_600.png`).

**Why no comments?** The site is statically deployed with no backend. If you want comments, mount Giscus or Cusdis as an island in `PostLayout.astro`.

**My OG image looks weird.** Run `cd content && npm run build` locally and inspect `dist/blog/<slug>/og.png`. The template lives at `content/src/lib/og.ts` вЂ” adjust the JSX-like tree to taste.

**The font fetch failed in CI.** The OG endpoint redirects to the static logo as a fallback. CI will still pass. Re-trigger after a few minutes вЂ” `fonts.googleapis.com` rarely has sustained outages.

**How is this different from the docs site?** `docs/` (VitePress) is product reference. Posts here are workflow guides, opinion, deep dives вЂ” the kind of content that earns citations.
